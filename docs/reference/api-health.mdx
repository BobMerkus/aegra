---
title: "Health & Info API"
description: "API reference for server health checks and service information."
---

Health and info endpoints are unauthenticated and always available. Use them for monitoring, load balancer probes, and service discovery.

## Root

<ParamField path="GET" type="/">
  Returns basic service identification.
</ParamField>

### Response

| Field | Type | Description |
|-------|------|-------------|
| `message` | `string` | Project name |
| `version` | `string` | Server version |
| `status` | `string` | Always `"running"` |

```bash
curl http://localhost:8000/
```

```json
{
  "message": "Aegra",
  "version": "0.5.3",
  "status": "running"
}
```

## Info

<ParamField path="GET" type="/info">
  Returns detailed service information and feature flags.
</ParamField>

### Response

| Field | Type | Description |
|-------|------|-------------|
| `name` | `string` | Service name |
| `version` | `string` | Server version |
| `description` | `string` | Service description |
| `status` | `string` | Always `"running"` |
| `flags` | `object` | Feature flag map |

Feature flags indicate which capabilities are enabled:

| Flag | Type | Description |
|------|------|-------------|
| `assistants` | `boolean` | Whether the assistants API is available |
| `crons` | `boolean` | Whether cron scheduling is available |

```bash
curl http://localhost:8000/info
```

## Health check

<ParamField path="GET" type="/health">
  Full health check that verifies all backend components.
</ParamField>

### Response

| Field | Type | Description |
|-------|------|-------------|
| `status` | `string` | `"healthy"` or `"unhealthy"` |
| `database` | `string` | Database connection status |
| `langgraph_checkpointer` | `string` | Checkpointer connection status |
| `langgraph_store` | `string` | Store connection status |

Returns `200` when all components are healthy, `503` when any component is unhealthy.

```bash
curl http://localhost:8000/health
```

```json
{
  "status": "healthy",
  "database": "connected",
  "langgraph_checkpointer": "connected",
  "langgraph_store": "connected"
}
```

Use this endpoint for comprehensive monitoring dashboards and alerts.

## Readiness probe

<ParamField path="GET" type="/ready">
  Kubernetes readiness probe. Returns `200` when the server is ready to accept traffic.
</ParamField>

### Response

Returns `{"status": "ready"}` with status code `200`, or `503` if the server is not ready.

```yaml
# Kubernetes deployment
readinessProbe:
  httpGet:
    path: /ready
    port: 8000
  initialDelaySeconds: 5
  periodSeconds: 10
```

---

## Liveness probe

<ParamField path="GET" type="/live">
  Kubernetes liveness probe. Returns `200` as long as the process is alive.
</ParamField>

### Response

Returns `{"status": "alive"}`.

```yaml
# Kubernetes deployment
livenessProbe:
  httpGet:
    path: /live
    port: 8000
  initialDelaySeconds: 15
  periodSeconds: 20
```

<Note>
  The liveness probe is a simple "am I alive?" check. Use `/ready` for traffic routing decisions and `/health` for full component status.
</Note>

## Error responses

All API endpoints use a standard error response format:

| Field | Type | Description |
|-------|------|-------------|
| `error` | `string` | Error type key (e.g. `"not_found"`, `"bad_request"`) |
| `message` | `string` | Human-readable error message |
| `details` | `object \| null` | Additional error context |

### Common HTTP status codes

| Code | Meaning |
|------|---------|
| `400` | Bad request — invalid input or missing required fields |
| `401` | Unauthorized — missing or invalid authentication |
| `403` | Forbidden — authenticated but lacking permissions |
| `404` | Not found — resource does not exist |
| `409` | Conflict — resource state conflict (e.g. deleting an active run) |
| `422` | Validation error — request body failed schema validation |
| `500` | Internal server error |
| `503` | Service unavailable — server is unhealthy or not ready |
